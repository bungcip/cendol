## Overview

The AST Dumper phase is responsible for taking the Annotated Abstract Syntax Tree (AST) and associated semantic information (Symbol Table, Type Table, Scope Manager) generated by the Semantic Analysis phase, and producing a human-readable and potentially AI-parsable output. Given the requirement for human readability in a browser, the primary output format will be HTML. This document details the design for generating a comprehensive HTML dump.

## Design Goals

*   **Human Readability**: The primary goal is to present the AST and semantic tables in a clear, intuitive, and easily navigable format for human developers using a web browser.
*   **Comprehensive Information**: Display all relevant information from the annotated AST nodes (type, value, span, semantic properties) and the semantic tables (symbol details, type definitions, scope hierarchy).
*   **Debuggability**: Facilitate quick identification of issues by providing clear visual cues, cross-referencing capabilities, and potentially interactive elements.
*   **Maintainability**: The dumper logic should be modular and easy to extend for future additions to the AST or semantic information.
*   **AI Parsability (Secondary)**: While primarily for humans, the HTML structure should be consistent enough to allow AI tools to extract information programmatically if needed.

## Inputs

The AST Dumper will receive the following outputs from the Semantic Analysis phase:

1.  **Annotated AST**: The root node of the Abstract Syntax Tree, where each node has been enriched with semantic information (e.g., resolved types, symbol references, constant values).
2.  **Symbol Table**: A complete table containing all declared symbols, their kinds, types, storage classes, scopes, and definition locations.
3.  **Scope Manager/Table**: Information about the scope hierarchy, including parent-child relationships and symbols defined within each scope.
4.  **Type Table**: A registry of all unique types encountered, including complex types like structs, unions, enums, and function types.
5.  **Source Code Mapping**: (Optional but highly recommended) A way to map source spans back to actual lines of code for displaying context.

## Outputs

The primary output will be a single HTML file (e.g., `output.html`) containing:

*   A structured, visual representation of the Annotated AST.
*   A tabular representation of the Symbol Table.
*   A tabular representation of the Scope Table.
*   A tabular representation of the Type Table.

## HTML Output Structure

The HTML file will be structured into distinct sections for each major component.

```html
<!DOCTYPE html>
<html>
<head>
    <title>C11 Compiler AST & Semantic Dump</title>
    <!-- CSS Styling will go here -->
</head>
<body>
    <h1>C11 Compiler AST & Semantic Dump</h1>

    <section id="ast-dump">
        <h2>Abstract Syntax Tree</h2>
        <!-- AST content here -->
    </section>

    <section id="symbol-table-dump">
        <h2>Symbol Table</h2>
        <!-- Symbol Table content here -->
    </section>

    <section id="scope-table-dump">
        <h2>Scope Table</h2>
        <!-- Scope Table content here -->
    </section>

    <section id="type-table-dump">
        <h2>Type Table</h2>
        <!-- Type Table content here -->
    </section>

    <!-- Optional JavaScript for interactivity -->
</body>
</html>
```

### 5.1. AST Representation

The AST will be represented using nested unordered lists (`<ul>` and `<li>` elements) to clearly show the hierarchical structure. Each `<li>` will represent an AST node.

*   **Node Information**: Each `<li>` will contain:
    *   `node-type`: The kind of AST node (e.g., `FunctionDefinition`, `BinaryExpression`, `Identifier`).
    *   `node-value`: The literal value or identifier name, if applicable.
    *   `span-info`: Source location (file, start line:col - end line:col).
    *   `semantic-info`: A block containing key semantic properties:
        *   `resolved_type`: The fully resolved type of the expression/declaration.
        *   `symbol_id`: A reference (e.g., a link or ID) to the corresponding entry in the Symbol Table.
        *   `is_lvalue`, `is_constant_expression`, `evaluated_value`, etc.
    *   Nested `<ul>` for child nodes.

**Example Snippet:**

```html
<ul class="ast-tree">
    <li>
        <span class="node-type">TranslationUnit</span>
        <span class="span-info">[main.c:1:1-20:1]</span>
        <ul>
            <li class="ast-node">
                <span class="node-type">FunctionDefinition</span>: <span class="node-value">main</span>
                <span class="span-info">[main.c:3:1-7:1]</span>
                <div class="semantic-info">
                    Type: <span class="semantic-type" data-type-id="type_int_void_func">int(void)</span>,
                    Symbol ID: <a href="#sym_main">sym_main</a>
                </div>
                <ul>
                    <li class="ast-node">
                        <span class="node-type">Identifier</span>: <span class="node-value">main</span>
                        <span class="span-info">[main.c:3:5-9]</span>
                        <div class="semantic-info">
                            Type: <span class="semantic-type" data-type-id="type_int_void_func">int(void)</span>,
                            Symbol ID: <a href="#sym_main">sym_main</a>
                        </div>
                    </li>
                    <!-- ... more children ... -->
                </ul>
            </li>
        </ul>
    </li>
</ul>
```

### 5.2. Symbol Table Representation

The Symbol Table will be presented as an HTML table (`<table>`).

*   **Columns**: `ID`, `Name`, `Kind`, `Type Info`, `Storage Class`, `Scope ID`, `Definition Location`, `Is Defined`, `Is Referenced`, `Is Completed`, `Details (JSON/String)`.
*   **Cross-referencing**: The `Type Info` column should link to the corresponding entry in the Type Table. The `Scope ID` column should link to the Scope Table. The `ID` column should have an anchor (`id="sym_..."`) for direct linking from AST nodes.
*   **Details**: Complex details (e.g., function parameters, struct members) can be presented as a compact JSON string or a more structured sub-table/list within the cell, or in a tooltip/modal via JavaScript.

**Example Snippet:**

```html
<table id="symbol-table">
    <thead>
        <tr><th>ID</th><th>Name</th><th>Kind</th><th>Type Info</th><th>Storage Class</th><th>Scope ID</th><th>Definition Location</th><th>Is Defined</th><th>Is Referenced</th><th>Details</th></tr>
    </thead>
    <tbody>
        <tr id="sym_main">
            <td>sym_main</td>
            <td>main</td>
            <td>Function</td>
            <td><a href="#type_int_void_func">int(void)</a></td>
            <td>extern</td>
            <td><a href="#scope_global">scope_global</a></td>
            <td>main.c:3:1</td>
            <td>True</td>
            <td>False</td>
            <td>{"is_definition": true, "parameters": []}</td>
        </tr>
        <!-- ... more symbols ... -->
    </tbody>
</table>
```

### 5.3. Scope Table Representation

The Scope Table will also be an HTML table.

*   **Columns**: `Scope ID`, `Parent Scope ID`, `Is Block Scope`, `Symbols in Scope`, `Types in Scope`.
*   **Cross-referencing**: `Parent Scope ID` links to other scope entries. `Symbols in Scope` lists symbol IDs, linking to the Symbol Table. `Scope ID` column will have an anchor (`id="scope_..."`).

**Example Snippet:**

```html
<table id="scope-table">
    <thead>
        <tr><th>Scope ID</th><th>Parent Scope</th><th>Is Block Scope</th><th>Symbols in Scope</th><th>Types in Scope</th></tr>
    </thead>
    <tbody>
        <tr id="scope_global">
            <td>scope_global</td>
            <td>-</td>
            <td>False</td>
            <td><a href="#sym_main">sym_main</a>, <a href="#sym_printf">sym_printf</a></td>
            <td><a href="#type_int">int</a>, <a href="#type_char_ptr">char*</a></td>
        </tr>
        <!-- ... more scopes ... -->
    </tbody>
</table>
```

### 5.4. Type Table Representation

The Type Table will be an HTML table, detailing all unique types.

*   **Columns**: `Type ID`, `Kind (e.g., Primitive, Pointer, Array, Struct, Union, Enum, Function)`, `Base Type (if applicable)`, `Qualifiers`, `Size`, `Alignment`, `Details (e.g., struct members, function parameters)`.
*   **Cross-referencing**: `Type ID` column will have an anchor (`id="type_..."`) for direct linking from AST nodes and Symbol Table entries. `Base Type` should link to other type entries.

**Example Snippet:**

```html
<table id="type-table">
    <thead>
        <tr><th>Type ID</th><th>Kind</th><th>Base Type</th><th>Qualifiers</th><th>Size</th><th>Alignment</th><th>Details</th></tr>
    </thead>
    <tbody>
        <tr id="type_int">
            <td>type_int</td>
            <td>Primitive</td>
            <td>-</td>
            <td>-</td>
            <td>4</td>
            <td>4</td>
            <td>{}</td>
        </tr>
        <tr id="type_int_void_func">
            <td>type_int_void_func</td>
            <td>Function</td>
            <td><a href="#type_int">int</a> (return)</td>
            <td>-</td>
            <td>-</td>
            <td>-</td>
            <td>{"parameters": []}</td>
        </tr>
        <tr id="type_struct_my_struct">
            <td>type_struct_my_struct</td>
            <td>Struct</td>
            <td>-</td>
            <td>-</td>
            <td>12</td>
            <td>4</td>
            <td>{"members": [{"name": "a", "type": "int"}, {"name": "b", "type": "char*"}]}</td>
        </tr>
        <!-- ... more types ... -->
    </tbody>
</table>
```

## 6. Styling (CSS)

A dedicated `<style>` block within the HTML `<head>` will provide basic styling for readability.

*   **Font**: Monospace for code-like appearance.
*   **Indentation**: Clear indentation for AST hierarchy.
*   **Color Coding**: Different colors for node types, values, semantic info, and span info.
*   **Borders/Backgrounds**: To visually separate sections and table rows.
*   **Collapsible elements**: (If JavaScript is implemented) Styling for collapsed/expanded states.

## 7. Interactivity (JavaScript - Optional)

While not strictly required for the initial dump, adding JavaScript can significantly enhance the debugging experience.

*   **Collapsible AST Nodes**: Allow users to expand/collapse branches of the AST.
*   **Search/Filter**: Implement search functionality for Symbol, Scope, and Type tables.
*   **Cross-referencing**: Clicking on a `symbol_id` in the AST jumps to the corresponding entry in the Symbol Table, and vice-versa. Similarly for types and scopes.
*   **Tooltips/Modals**: Display detailed information for complex entries (e.g., struct members, function signatures) on hover or click.
*   **Source Code View**: If source code mapping is available, clicking a span could highlight the relevant code in a separate pane or popup.

## 8. Error Handling

The dumper itself should be robust to malformed input from previous phases.

*   **Graceful Degradation**: If an AST node or semantic entry is incomplete or malformed, the dumper should still attempt to display as much information as possible, perhaps with a warning.
*   **Clear Error Messages**: If the dumper encounters an internal error, it should log it clearly.

## 9. Performance Considerations

*   **Large Files**: For very large source files, the generated HTML could become very big. Consider:
    *   **Lazy Loading**: (With JavaScript) Only load/render parts of the AST as they are scrolled into view.
    *   **Pagination**: (For tables) Implement pagination for Symbol, Scope, and Type tables.
*   **Efficient HTML Generation**: The Rust implementation should use efficient string building or HTML templating libraries to minimize overhead.

## 10. Integration with Compiler Pipeline

The AST Dumper will be the final phase of the C11 compiler pipeline.

*   It will receive the `SemanticResult` struct (or similar) as its input.
*   It will output the generated HTML content to a specified file path.
*   The `DumpConfig` struct (as defined in `c11_compiler_design.md`) will be extended to include HTML-specific options (e.g., `pretty_print_html`, `include_javascript_interactivity`).
